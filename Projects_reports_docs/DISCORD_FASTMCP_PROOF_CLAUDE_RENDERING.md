# Claude-side HTML Rendering — Proof of Concept & Reference
**Date :** 2026-05-16  
**Pour :** Discord FastMCP #help / #general  
**Contexte :** Alternative à PrefabApp pour render dashboard dans Claude Desktop

---

## TL;DR

Claude Desktop can render rich interactive HTML dashboards **natively**, without PrefabApp, without iframes, without handshake. The rendering is handled by Claude's own UI layer (the Visualizer tool), triggered by the MCP tool response data. The MCP server stays pure data — Claude does the presentation.

---

## How it works — mechanism

```
MCP Tool (Python)          Claude Desktop           User sees
─────────────────          ──────────────           ─────────
server_health()    ──►     receives JSON    ──►     Claude calls
nmap(target)       ──►     parses output    ──►     show_widget()
whatweb(url)       ──►     extracts data    ──►     with HTML built
                                                    from tool data
                                            ──►     dashboard renders
                                                    inline in chat
```

**Key insight :** Claude has a built-in `show_widget(html)` capability. When Claude receives MCP tool output, it can use that data to build and render HTML inline — no PrefabApp, no extra transport, no CSP issues.

This is not a hack. It is the documented behavior of Claude's artifact/widget system, available in Claude Desktop and claude.ai.

## Minimal POC — reproduce it yourself

Paste this into any Claude Desktop conversation after connecting a FastMCP server :

**Step 1 — MCP tool (Python)**
```python
from mcp.server.fastmcp import FastMCP

mcp = FastMCP("demo")

@mcp.tool()
def get_demo_data() -> dict:
    return {
        "target": "example.com",
        "ip": "93.184.216.34",
        "ports": [
            {"port": 80, "service": "http", "state": "open"},
            {"port": 443, "service": "https", "state": "open"},
        ],
        "web_tech": "nginx/1.24",
        "status": "healthy"
    }

if __name__ == "__main__":
    mcp.run()
```

**Step 2 — Prompt to Claude Desktop**
```
Call get_demo_data(), then render an HTML dashboard showing 
the target info, open ports as a table, and server health metrics.
Use inline CSS only, no external dependencies.
```

Claude will call the tool, receive the JSON, and render an HTML widget inline. No PrefabApp needed.

## Why this works better than PrefabApp (for our use case)

| | PrefabApp | Claude-side rendering |
|---|---|---|
| Iframe/handshake | required | none |
| CSP issues | yes (known) | none |
| stdio vs HTTP | matters | irrelevant |
| Data freshness | static prefab | always from live tool data |
| Interactivity | limited to prefab widgets | full HTML/JS |
| Debug complexity | high | low (just JSON + prompt) |
| Claude Desktop reliability | inconsistent | consistent |

---

## Limitations to be honest about

- **Claude controls the render** — the HTML is generated by Claude, not your server. You cannot ship pixel-perfect branded UI this way.
- **Not deterministic** — Claude may render slightly differently across sessions. For strict UI requirements, PrefabApp is still the right answer.
- **Token cost** — Claude generates the HTML each time. For very large dashboards (10+ panels) this adds latency.
- **No persistent state** — the widget lives in the conversation. Refresh = re-call the tools.
