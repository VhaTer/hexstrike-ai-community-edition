## 2. async get_pulse_data + asyncio.gather

`pulse_app.py` :
- `get_pulse_data()` converti de `def` → `async def`
- 6 sub-calls lourds lancés en parallèle via `asyncio.gather()` + `asyncio.to_thread()`
- `_safe_call(fn, fallback)` helper — try/except par sub-call
- Latence : ~300ms sequential → ~100ms parallel

### B1 — déplacer get_overview() dans le gather

`get_overview()` contient `psutil.cpu_percent(interval=0.1)` qui bloque 100ms.
Avant S30b : appelé **avant** le gather (séquentiel).
Après S30b : ajouté comme sub-call dans le gather (parallèle).
Gain : ~105ms retirés du chemin critique.

### B5 — pré-warming _get_parameter_optimizer()

Ajouté `eng._get_parameter_optimizer()` dans `_prewarm_singletons`.
Premier plan call ne paye plus 30-50ms de lazy init.

## 5. Split tools — remplacer get_pulse_data()

`get_pulse_data()` retiré. 8 tools individuels `@app.tool()` exposés :

| Tool | Description | Taille réponse |
|------|-------------|----------------|
| `get_overview()` | Version, uptime, RAM, tools, status | ~1KB |
| `get_scope(target)` | Actif + type detection | ~1KB |
| `get_surface(target)` | Ports, services, techs, risk | ~2KB |
| `get_findings(target)` | Vulns nuclei/nikto | ~2KB |
| `get_plan(target, objective)` | Attack chain steps | ~3KB |
| `get_active_tools()` | Processes, workers, queue | ~1KB |
| `get_history(target, limit)` | Scan history filtré | ~2KB |
| `get_tool_intelligence()` | Baseline vs live scores | ~3KB |

**Avantages** :
- Chaque réponse <5KB, <100ms au lieu de ~30KB, ~300ms
- Le modèle appelle ce dont il a besoin, pas tout à chaque fois
- Impressions de raisonnement pas-à-pas pour l'utilisateur

**Nettoyage** :
- `get_recent_scans()` supprimé (remplacé par `get_history(target)`)
- `_guess_target_type()` n'est plus `@app.tool()` (helper interne)
- `asyncio` import retiré
- `_safe_call()` retiré
- `get_pulse_data` retiré de `always_visible` dans `server_setup.py`
- `recent_scans` retiré du state dict de `pulse_dashboard()` (dead key)
